# Architecture

## How the system works

The bot places outbound calls via Twilio and uses OpenAI's Realtime API to simulate a patient having a natural voice conversation. When a call is initiated, Twilio dials the test number and simultaneously opens a WebSocket media stream to our FastAPI bridge server. The bridge relays audio in both directions: audio arriving from the PGAI agent is forwarded to an OpenAI Realtime session, and the audio OpenAI generates in response is sent back through Twilio to the agent. Because both Twilio and OpenAI Realtime support G.711 µ-law (8 kHz) natively, no format conversion is needed — bytes flow straight through. OpenAI's server-side VAD handles turn detection automatically, so the patient waits for the agent to finish speaking before responding, just like a real caller would. After the call ends, the bridge writes a full two-sided transcript (the agent's speech via Whisper transcription, the patient's speech via Realtime audio transcript events), and a separate analyzer pass sends that transcript to GPT-4o to flag any agent bugs against the expected behavior for each scenario.

## Key design choices

Using the OpenAI Realtime API as the patient voice was the most direct path to the evaluation criterion that matters most: a lucid, natural-sounding conversation. A pipeline of separate STT → LLM → TTS components would add latency at every step and create awkward pauses. Realtime collapses all three into a single WebSocket session, which keeps turn-taking responsive. Twilio was chosen for telephony because it provides outbound calling, call recording (mp3), and media streaming in one API, and has a well-documented pattern for bridging to OpenAI Realtime. The bridge runs as a lightweight FastAPI server started locally with ngrok providing the public endpoint, which avoids cloud deployment complexity during development. Scenarios are defined as plain Python dataclasses with a system prompt and an expected-behavior field; the same expected-behavior text is fed to the post-call analyzer so bugs are evaluated against a clear criterion rather than a vague quality rubric.
